{# ── Step 2: the chosen mode's details ───────────────────────────────── #} {# MOVE #}

Move {{ detail.domain }} to another node

What Hyperion does
  1. Takes a snapshot on the source, transfers it to the target, and restores it there.
  2. Opens a live progress page so you can follow each step.
  3. Suspends the source once the target is up.
What you do next
  1. Point the domain's DNS at the target node's IP.
  2. On the target, re-issue the TLS certificate from the SSL tab. Private keys never travel between nodes.
  3. Once you've confirmed the site works, delete the suspended original from its Danger tab.
{# COPY #}

Copy {{ detail.domain }} to a new domain

What Hyperion does
  1. Creates a duplicate at the new domain, with its own files and database.
  2. Opens a live progress page so you can follow along.
  3. For WordPress sites, rewrites the URLs to the new domain automatically.
  4. If DNS already points to the target, issues a TLS certificate for the new domain.
  5. Leaves the original running, completely untouched.
{# EXPORT #}

Export a transfer file

Use this when the destination isn't in this cluster — an off-cluster server, or one you only reach over SSH. Hyperion builds a single transfer file (site files, database, and a manifest); you move it across and import it on the other side.

Then, on the destination
  1. The result page gives you a one-click import link and token, or an scp plus hctl one-liner.
  2. To pull the transfer file into another Hyperion node, open Import hosting from another node there.
  3. Re-point DNS and re-issue the certificate on the destination. Private keys never travel between nodes.
