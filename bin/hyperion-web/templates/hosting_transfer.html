{# ── Step 2: the chosen mode's details ───────────────────────────────── #} {# MOVE #}

Move {{ detail.domain }} to another node

What happens
  1. Snapshot taken on the source, transferred to the target, restored there.
  2. You're taken to a live progress page (step-by-step).
  3. The source is suspended once the target is up.
Then it's up to you
  1. Point the domain's DNS at the target node's IP.
  2. On the target, re-issue the TLS certificate (SSL tab) — keys never travel.
  3. Once you've confirmed the site works, delete the suspended original from its Danger tab.
{# COPY #}

Copy {{ detail.domain }} to a new domain

What happens
  1. A duplicate is created at the new domain (its own files + database).
  2. You're taken to a live progress page.
  3. For WordPress sites the URLs are rewritten to the new domain automatically.
  4. The original keeps running, completely untouched.
{# EXPORT #}

Export a transfer file

Use this when the destination isn't in this cluster (an off-cluster server, or one you only reach over SSH). Hyperion makes a single bundle (site files + database + a manifest); you move it across and import it on the other side.

Then, on the destination
  1. The result page gives you a one-click import link + token, or an scp + hctl one-liner.
  2. To pull it into another Hyperion node, use Hostings → Import a site there.
  3. Re-point DNS and re-issue the certificate on the destination — private keys never travel.
