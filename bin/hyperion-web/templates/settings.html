How outbound mail works

  • Relay host — pick a real provider (Postmark, Mailgun, SendGrid, Brevo, AWS SES, Gmail SMTP) or point at a local postfix that's been configured to relay outbound. Direct-from-VPS sends to public mailboxes almost always land in spam.
  • SPF — the recipient's mail server checks if your relay's IP is on the From-address domain's permitted-sender list. Set up via a DNS TXT record on your domain; Hyperion shows per-hosting SPF analysis on each hosting's detail page.
  • Sender address — must be a domain you control AND whose SPF includes your relay. Auto-detect tries to derive a sensible default from /etc/mailname.
  • Per-node — each worker has its own [email] section in its own agent.toml. Pick a node in the From dropdown above to verify that node's outbound path.
  • Every send is logged at /emails with the full SMTP response — so you can tell "queued" (250) from "rejected by relay" (550) without leaving the UI.

How HTTPS issuing works

  • Each hosting starts with a self-signed cert right after provisioning — enough for nginx to come up.
  • When http://<domain> resolves to this server, you can issue a real Let's Encrypt cert from the hosting's detail page → SSL tab.
  • Hyperion uses HTTP-01 (file under {{ config.acme.challenge_dir }}/<token>) — no DNS API integration needed.
  • Certs auto-renew 30 days before expiry; failures alert via Slack / email.
  • Per-hosting contact email overrides this one on the hosting detail page.

{# ========================================================== BACKUPS — remote target + retention policy ========================================================== #}

Off-site backup target

{% if config.backup_remote.enabled %}pushing {% else %}local only{% endif %}

Backups always land on local disk first (under /var/lib/hyperion/backups/local). Optionally Hyperion can also push a copy to an off-site FTP/FTPS/SFTP server immediately after the local backup finishes.

Retention policy

Backups older than max age are pruned after every new backup completes, but at least keep latest N per hosting are always retained (so an old hosting that nobody touches for months still has the last few backups around when you need them).

Cluster placement

Uncheck this if the master should be a control plane only and all hostings live on workers.
With this off, the Target-node dropdown on /hostings/new hides the master option, and creating directly via hctl on the master will be refused. Existing hostings on the master stay where they are — the toggle only affects new creates.

Test nodes & staging sites

Mark some nodes as test-only for staging / preview work. Hostings created on a test node get an auto-generated subdomain (so you can't accidentally squat a real customer's domain there), and the wizard refuses to put a production domain on a test node (and vice versa).

Tip: the easier way to mark a node as test is on /install — each row has a one-click toggle. This card is the manual fallback if you want to bulk-edit or paste a CSV.

The IDs from /install column "ID". Empty = test-node feature is off (every node is treated as production).

Two tokens get substituted: {name} = the short name the operator types in the wizard, {node} = the target node's ID. Result is lower-cased.

Example. Template {name}.{node}.staging.example.com, operator types blog in the wizard, target node is lab-01 → final domain blog.lab-01.staging.example.com.
Make sure you have a wildcard DNS record (e.g. *.staging.example.com → your test-node IP) so the auto-generated subdomains resolve.

Sets blog_public = 0 in WordPress right after a fresh install on a test node — Google & friends are asked not to index. Operator can flip it later in WP admin → Settings → Reading.

Trash / soft-delete

When on, deleting a hosting moves it to /trash: nginx serves 503, FPM stops, DB user is locked, OS user is locked, but files / DB / certs / user are preserved. After the retention window the scheduler runs the hard-delete pipeline automatically. Off (the default) = immediate hard-delete as before.

The scheduler hard-deletes any trashed hosting older than this. Operators can also Restore (un-trash) or Delete permanently from /trash before the timer expires.

Multi-node tips

  • Nodes & install page lists enrolled workers, lets you mint invite tokens for new ones, run a connectivity test, and trigger remote apt/hyperion updates.
  • Updates — pick any enrolled node from /install → click Update… → tick System packages and/or Hyperion → log streams live in the row.
  • Stats at /stats default to the cluster total: hostings, disk and bandwidth summed across master + all workers. Switch the dropdown to a node to see per-node sparklines.
  • Service health at /services shows each node's systemd units. Click Install to apt-install a missing service — output streams live (no more "dpkg returned an error" with no detail).
