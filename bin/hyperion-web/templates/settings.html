How outbound mail works

  • Relay host — pick a real provider (Postmark, Mailgun, SendGrid, Brevo, AWS SES, Gmail SMTP) or point at a local postfix configured to relay outbound. Sending directly from a VPS to public mailboxes almost always lands in spam.
  • SPF — the recipient's mail server checks whether your relay's IP is on the From-address domain's permitted-sender list. Set this up with a DNS TXT record on your domain; Hyperion shows per-hosting SPF analysis on each hosting's detail page.
  • Sender address — must use a domain you control whose SPF includes your relay. Detection derives a sensible default from /etc/mailname.
  • Per-node — each worker has its own [email] section in its own agent.toml. Pick a node in the From dropdown above to verify that node's outbound path.
  • Every send is logged at Emails with the full SMTP response, so you can tell "queued" (250) from "rejected by relay" (550) without leaving the panel.

How HTTPS issuing works

  • Each hosting starts with a self-signed certificate right after provisioning — enough for nginx to come up.
  • Once http://<domain> resolves to this server, you can issue a real Let's Encrypt certificate from the hosting's detail page → SSL tab.
  • Hyperion uses the HTTP-01 challenge (a file under {{ config.acme.challenge_dir }}/<token>), so no DNS API integration is needed.
  • Certificates auto-renew 30 days before expiry; failures alert via Slack or email.
  • A per-hosting contact email, set on the hosting detail page, overrides this one.

Backups always land on local disk first, under /var/lib/hyperion/backups/local. Hyperion can also push a copy to an off-site FTP/FTPS/SFTP server as soon as the local backup finishes.

Retention policy

After every new backup completes, backups older than max age are pruned — but at least keep latest N per hosting are always retained. So a hosting nobody has touched for months still keeps its last few backups for when you need them.

Cluster placement

Turn this off to make the master a control plane only, with every hosting living on a worker.
While off, the target-node dropdown in the create wizard hides the master option, and creating directly with hctl on the master is refused. Existing hostings on the master stay put — this toggle affects new creates only.

Security

When on, an Administrator or Owner who signs in without 2FA enrolled is sent straight to the enrollment card and can't reach the rest of the panel until they set it up. Operators and read-only users are unaffected. Turn it off only briefly, for example during a server move. Applies within about 30 s; no restart needed.

When on, the master refuses any RPC to a worker whose TLS certificate doesn't match the SPKI pin that worker reported on its heartbeat. This hardens the master→worker channel against a man-in-the-middle. Turn this on only after the per-node pins on Nodes show 🔒 TLS pin on file and you have seen no warn-only pin-mismatch warnings in the logs: a stale pin blocks management of that worker until its next heartbeat. Workers that haven't reported a pin yet are never blocked. Applies within about 30 s; no restart needed.

Multi-node tips

  • The Nodes page lists enrolled workers, mints invite tokens for new ones, runs a connectivity test, and triggers remote apt and Hyperion updates.
  • Updates — on the Nodes page, pick any enrolled node, click Update…, tick System packages and/or Hyperion, and the log streams live in the row.
  • Stats at Stats default to the cluster total: hostings, disk, and bandwidth summed across the master and all workers. Switch the dropdown to a node for its own sparklines.
  • Service health at Services shows each node's systemd units. Click Install to apt-install a missing service; the output streams live, with the full error instead of a bare "dpkg returned an error".

Trash / soft-delete

When on, deleting a hosting moves it to Trash: nginx serves 503, FPM stops, and the database and OS users are locked, but the files, database, certificates, and user are preserved. After the retention window, the scheduler runs the permanent-delete pipeline automatically. When off (the default), deletes are permanent immediately, as before.

{# Tell post_config to land us back on /settings#retention instead of the section→tab default (which would send us to #cluster). #}

The scheduler permanently deletes any trashed hosting older than this. Before the timer expires, you can still Restore a hosting or Delete it permanently from Trash.

Audit log retention

The audit log records every state-changing operation as a tamper-evident BLAKE3 hash chain. Without retention it grows indefinitely — on a busy cluster, tens of MB per month. Set a window in days below to purge older entries automatically; the chain re-anchors at the oldest surviving row, so verify_chain still works on what's left. 0 (the default) keeps everything forever.

Typical values: 90 for compliance-light setups, 365 for tighter rules, or 0 if you export /audit?export=csv off-cluster and want everything kept locally. Purges run on the normal scheduler tick (about every 60 s); no extra cron job is needed.
